## 在流程中的位置

`GET /api/v1/verify` 是**购买验证端点**——由开发者后端用于强制执行支付。它在生命周期中出现在两个点上：

```
A 点（支付前）：
  Agent 请求资源 → 开发者后端调用 GET /api/v1/verify
    → purchased=false → 向 Agent 返回 HTTP 402

B 点（支付后）：
  Agent 重试资源 → 开发者后端调用 GET /api/v1/verify
    → purchased=true → 返回 200 + 结果
```

<Warning>
  **这是防止支付被绕过的首要防线。** 没有后端验证，用户可以编辑 `skill.md` 删除支付代码块。对于任何有实际定价的技能，务必在服务端验证购买状态后再提供资源。
</Warning>

## 查询参数

| 参数 | 必需 | 类型 | 描述 |
|-----------|----------|------|-------------|
| `agent_did` | 是 | string | 买家/Agent DID（`did:solana:...`） |
| `skill_did` | 是 | string | 你的开发者 DID——被检查的技能（`did:solana:...`） |

## 认证

此端点支持两种认证方式：

| 方式 | 请求头 | 最适合 |
|--------|--------|----------|
| **API Key** | `X-API-Key: <key>` | 开发者后端 → StablePay（推荐） |
| **DID 签名** | `X-StablePay-DID` + `X-StablePay-Signature` + `X-StablePay-Timestamp` + `X-StablePay-Nonce` | Agent 通过插件查询 |

<Info>
  推荐开发者后端使用 API Key 认证，因为无需每次请求都用钱包签名。从 StablePay 平台获取你的 API Key。
</Info>

## 响应

### purchased = true

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "purchased": true,
    "purchase_time": "2026-05-29T10:30:00Z",
    "tx_id": "uuid-of-payment-transaction",
    "amount": "1.00",
    "currency": "USDC"
  }
}
```

### purchased = false

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "purchased": false
  }
}
```

<Note>
  `true` 和 `false` 两种结果都返回 HTTP 200，`code: 0`。你需要检查 `data` 中的 `purchased` 字段——**而非** HTTP 状态码或顶层 code。
</Note>

## 异步延迟

购买记录通过 RocketMQ **异步**写入：

```
Payment Service → RocketMQ（payment_events topic）→ Verification Service → MySQL（purchase_records）
```

时间：
- **正常**：`POST /api/v1/pay` 返回后 < 5 秒
- **边缘情况**：高负载下最多 30 秒

<Warning>
  `POST /api/v1/pay` 成功后，购买记录可能**不会立即可见**。StablePay 插件通过最多 6 次重试（间隔 1.5 秒）来处理此问题。如果你构建自定义客户端，请实现类似的重试逻辑。
</Warning>

## 典型后端流程

```
func HandleRequest(w http.ResponseWriter, r *http.Request) {
    agentDid := r.URL.Query().Get("agent_did")   // 来自 Agent 请求
    mySkillDid := "did:solana:AbCdEf..."          // 你的开发者 DID

    // 检查购买——每次请求都要调用
    purchased := verifyPurchase(agentDid, mySkillDid)

    if !purchased {
        // 返回 402 及支付挑战
        w.WriteHeader(402)
        json.NewEncoder(w).Encode(buildX402Challenge(mySkillDid, "1.00"))
        return
    }

    // 已购买——提供资源
    w.Write(executeSkillLogic())
}
```

## 错误处理

| 错误码 | 消息 | 原因 | 恢复方式 |
|------|---------|-------|----------|
| `10001` | invalid parameters | 缺少 `agent_did` 或 `skill_did` | 确保两个查询参数都已填写 |
| `10003` | permission denied | API Key 或 DID 签名无效或缺失 | 检查认证头 |
| `10004` | signature verification failed | DID 签名已过期或格式错误 | 刷新时间戳和 nonce，重新签名 |

## 批量替代方案

如果你需要检查同一 Agent 的多个技能（例如仪表盘显示所有技能的购买状态），使用 `POST /api/v1/verify/batch` 而非逐个调用此端点。
